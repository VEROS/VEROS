163  // Data structures for handling DSR calls. We implement two DSR
164  // handling mechanisms, a list based one and a table based
165  // one. The list based mechanism is safe with respect to temporary
166  // overloads and will not run out of resource. However it requires
167  // extra data per interrupt object, and interrupts must be turned
168  // off briefly when delivering the DSR. The table based mechanism
169  // does not need unnecessary interrupt switching, but may be prone
170  // to overflow on overload. However, since a correctly programmed
171  // real time application should not experience such a condition,
172  // the table based mechanism is more efficient for real use. The
173  // list based mechainsm is enabled by default since it is safer to
174  // use during development.

231  // Interrupt disable data. Interrupt disable can be nested. On
232  // each CPU this is controlled by disable_counter[cpu]. When the
233  // counter is first incremented from zero to one, the
234  // interrupt_disable_spinlock is claimed using spin_intsave(), the
235  // original interrupt enable state being saved in
236  // interrupt_disable_state[cpu]. When the counter is decremented
237  // back to zero the spinlock is cleared using clear_intsave().
238 
239  // The spinlock is necessary in SMP systems since a thread
240  // accessing data shared with an ISR may be scheduled on a
241  // different CPU to the one that handles the interrupt. So, merely
242  // blocking local interrupts would be ineffective. SMP aware
243  // device drivers should either use their own spinlocks to protect
244  // data, or use the API supported by this class, via
245  // cyg_drv_isr_lock()/_unlock(). Note that it now becomes
246  // essential that ISRs do this if they are to be SMP-compatible.
